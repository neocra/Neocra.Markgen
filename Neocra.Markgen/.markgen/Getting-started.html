Hello Be:Mo is a user-centric MSP (Mobility Services Provider) that allows developpers to build Mobile, InCar or WebApp custom applications. A series of APIs are available to enable the developpers to control all the stages of their final User experience.

This 'Getting started' section aims to provide insights into the key features enabled by Hello Be:Mo API Platform and how to put them into action.

The section is divided into the followng subsections :

  1. Prerequisite 1 - Authentication
  2. Prerequisite 2 - Create a User
  3. Journey - Detection & Information
  4. Digital charging process - Insight
  5. Quick start - Digital charging session
  6. RFID charging process - Insight

  8. Quick start - Mock RFID session

API - 'Create a user' in detail

create a user

This reference describes how to create a new user through the REST API. At a minimum, you must specify the required properties for the user. You can optionally specify any other writable properties (optional parameters).

Required parameters

civility : The contact's civility ("Ms" for a lady or "Mr" for a gentleman)

firstname : The contact's first name

lastname : The contact's last name

Optional parameters

email : The contact's email address. Limitation : Be:Mo doesn't realize unicity checks on email entries.

billingAddress refers to the place to which the invoice is sent. It is also specified on the invoice.
Limitation : To set up a billing address, you must provide , at a minimum, values to line1, city, zipCode and country fields.

  • line1 : The first line of the contact's address, which is often a street address or business name.
  • line2 : The second line of the contact's address.
  • line3 : The third line of the contact's address.
  • line4 : The fourth line of the contact's address.
  • city : The city of the contact's address.
  • localityRegionState : The locality, region or state of the contact's address.
  • zipCode : The zipcode of the contact's address.
  • country : The country of the contact's address.

shippingAddress refers to the place to which the RFID card should be delivered. Limitation : To set up a shipping address, you must provide , at a minimum, values to line1, city, zipCode and country fields.

  • line1 : The first line of the contact's address, which is often a street address or business name.
  • line2 : The second line of the contact's address.
  • line3 : The third line of the contact's address.
  • line4 : The fourth line of the contact's address.
  • city : The city of the contact's address.
  • localityRegionState : The locality, region or state of the contact's address.
  • zipCode : The zipcode of the contact's address.
  • country : The country of the contact's address.

postalAddress refers to the place to which the communication letters (excepted RFID card and invoices) should be sent. Limitation : To set up a postal address, you must provide , at a minimum, values to line1, city, zipCode and country fields.

  • line1 : The first line of the contact's address, which is often a street address or business name.
  • line2 : The second line of the contact's address.
  • line3 : The third line of the contact's address.
  • line4 : The fourth line of the contact's address.
  • city : The city of the contact's address.
  • localityRegionState : The locality, region or state of the contact's address.
  • zipCode : The zipcode of the contact's address.
  • country : The country of the contact's address.

consentDate : The date of expression of consent. It often refers to the date at which the user has agreed to the Terms & Conditions.

externalId : You can set up an externalId, which is often the userId in your own system. Once this value registered, you can retrieve the users created by using the endpoint GetUser (by externalId) Limitation : Be:Mo doesn't realize unicity checks on externalId entries.

externalDatas : You can freely setup and store a collection of externalDatas, meaningful in your business context. Limitation : Be:Mo doesn't use those datas in its processes.

Important outputs

userId : identifier of user in Be:Mo's database.

isActive : activation status of user. True (default state) if user is active, false if not.

API - 'Get clustered places around a given position' in detail

Get clustered places around a given position

Retrieves the nearest POI around a given location, grouped into clusters, according to given criterias (search radius, desired product, distribution network...). Depending on the zoom level (from 1 to 12), this endpoint enables to visualize on a map a reduced number of clusters instead of tens, hundreds or thousands single points.

Required parameters

userId : user's identifier in Be:Mo's data base. contained in the callout of the endpoint create User (Prerequisite #2)

Optional parameters

zoom : zoom level value (from 1 to 12). If no value is entered, zoom = 1 (the wider zoom range) is set by default.

vehicleId (optional): vehicle's identifier in Be:Mo's data base. contained in the callout of the endpoint Create a vehicle

Longitude : longitude coordinate of User location (current position of the User device or entered manually)

Latitude : latitude coordinate of User location (current position of the User device or entered manually)

productCodes : Be:Mo uses codes to distinguish the different products types from each other. It can refer to :

  • for ICE and PHEV, the fuel types. Example : E85 for Ethanol, B7 for Diesel etc.
  • for PEV and PHEV, the attributes of the energy supply (range of power delivered, speed of charge). Example : E_Standard for a Slow charging (> 5h), E_ultraFast for an Ultra fast charging (< 30 min) etc. The complete list of productCodes can be retrieved through the endpoint Get Products List

connectorCodes : Be:Mo uses codes to distinguish the different charging connectors (plug and socket) from each other. The complete list of connectorCodes can be retrieved through the endpoint Get Connectors List Limitation : This field only makes sense for Electric (PEV) and Plug-in hybrid vehicles (PHEV) .

Distance : search radius in kilometers (km)

PlaceNetworkCode : network/brand of POI Example : Aral , BP, Eni or Shell for fuel service stations.

AvailableProduct : True to filter the results and return only available products per place.

DigitalJourney : True to filter the results and return the places enabling digital journey (through a third-party application).

latitude & latitude : GPS coordinates of the barycenter of all places grouped into the cluster

count : number of places/stations grouped into the cluster.

When mode = places :

placeId : identifier of the place in the POI's database.

latitude & latitude : GPS coordinates of the place

attributes : list of capabilities & connectors provided by the place.

list of productIddelivered by the place

other general information (station name, address, radial distance from entered location to place...)

Digital charging process - Insight

Digital Charging process in real-life

Limitations

  1. The digital (or remote) session IS ONLY available in a MINORITY of places. The places enabling this feature are recognisable by the tag capability.remote_start_stop_capable in the response payloads of Detection (Get clustered places around [...}, Get places around [...]) and Information (Get place details) endpoints.
  1. Be:Mo can only forward session (and session states) updates, through event messages, if we receive them first from the CPO. The entire sequence of final user notification depends on the reliability of the CPO.

Process

START A DIGITAL SESSION

Process steps in detail
Step number Triggered by... (Sender) For ... (Recipient) In order to... (Action) Through... (Medium)
1 Final user Partner Trigger a session start User device
2 Partner Be:Mo Start a Digital (or Remote) session API call
3 Be:Mo CPO Forward session start order API call
4 CPO Be:Mo Aknowledge ‘session start’ order & create session context (sessionId, state = 'pending') while waiting for user action API (callback of 3)
5 Be:Mo Partner Forward ‘Success’ of CPO acknowledgment in synchronous mode API (callback of 2)
6 Partner Final user Notify the final user User device
6bis CPO Final user Notify the final user with a visual signal on the EVSE & unlock the connector EVSE (physical equipment)
7 Final user CPO Plug his/her car & ensure that the charging session is in progress EVSE (physical equipment)
8 CPO Be:Mo Update session state (from ‘pending’ to ‘active’) Event message
9 Be:Mo Partner Forward session update (from ‘pending’ to ‘active’) via the event SessionActive Event message
10 Partner Final user Notify the final user User device

RFID charging process - Insight

RFID Charging process in real-life

Limitations

  1. The RFID session process is entirely triggered and controlled by the CPO. Be:Mo is only notified afterwards by event messages.
  1. Be:Mo can only forward session (and session states) updates, through event messages, if we receive them first from the CPO.
  1. Even more than Digital session process, the RFID journey is entirely dependent on the reliability of the CPO.

Process

START A RFID SESSION

Process steps in detail
Step number Triggered by... (Sender) For ... (Recipient) In order to... (Action) Through... (Medium)
1 Final user CPO Trigger the session directly by using his/her RFID on an EVSE equipped with RFID reader RFID card + EVSE (physical equipment)
2 CPO Final user Notify the final user with a visual signal on the EVSE & unlock the connector EVSE (physical equipment)
3 Final user CPO Plug his/her car & ensure that the charging session is in progress EVSE (physical equipment)
4 CPO Be:Mo Aknowledge ‘session start’ order & create session context (sessionId, state = 'pending') while waiting for user action Event message (consequence of 2)
5 Be:Mo Partner Forward ‘Success’ of CPO acknowledgment in asynchronous mode via the SessionPending event (sessionId, state = ‘pending’) Event message
6 Partner Final user Notify the final user User device
7 CPO Be:Mo Update session context (sessionId, state = 'active') once the car is successfully charging Event message (consequence of 3)
8 Be:Mo Partner Forward session update (from ‘pending’ to ‘active’) via the SessionActive event Event message
9 Partner Final user Notify the final user User device

Quick start - Provide RFID cards to your final users

The RFID ( Radio Frequency IDentification) Card, is a physical card, linked to a User account, which allows the driver to access securely the Charging network and the services it provides. Nearly all the Charging pools connected with Be:Mo's infrastructure have RFID card readers that enable, by touching the card on & off, to :

  • Recognize an active card,
  • Authenticate the related User account
  • Trigger & complete the Charging process (from session start until transaction receipt, including payment).

Be:Mo system enables you to implement 2 different approaches to provide RFID cards to your final users :

Implementation 1 - Order a RFID Card through Hello Be:Mo system Be:Mo (associated with a main European secured cards manufacturer) manages the end-to-end process of card ordering. Through our Platform you can initiate the order (on behalf of your clients), monitor its progress (until delivery) and control the edited card lifecycle.

Implementation 2 - Enroll external RFID card into Hello Be:Mo system You already have your own RFID card sourcing process. Through our Platform you can enroll your card as Be:Mo material and then, control its lifecycle.

Quick start - Mock RFID session

To mimick real-life RFID sessions, in Test environments only, Be:Mo provides the Mock session microservice. Thanks to this artefact, a Dev Team can mock a dozen of use cases (OK and KO ) which cover a representative sample of real situations. The Team will be able to anticipate :

  • all the session status that its system could face
  • the inability for several CPO to send all or part of update notifications
  • the failure cases etc.
